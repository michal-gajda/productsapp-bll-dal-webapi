# Coding Standards & Architecture Guidelines

## 🎯 Project Overview
- **Architecture**: 3-Layer (WebAPI → BLL → DAL)
- **Framework**: .NET 10
- **Pattern**: Repository Pattern with Dependency Injection
- **Mapping**: Explicit Operators (no separate mapper classes)

---

## ❌ FORBIDDEN PRACTICES

### 1. Primary Constructors in Records
**ZAKAZANE** - Primary constructors dla records są **ZABRONIONE**.

```csharp
// ❌ NIGDY NIE UŻYWAJ
public record ProductResponse(int Id, string Name, decimal Price);

// ✅ ZAWSZE UŻYWAJ
public record ProductResponse
{
    public required int Id { get; init; }
    public required string Name { get; init; }
    public required decimal Price { get; init; }
}
```

**Powody:**
- Git diff pokazuje zmiany w wielu liniach (trailing comma problem)
- Atrybuty walidacji są nieczytelne
- Trudne debugowanie
- Merge conflicts przy równoległej pracy

---

### 2. One-Line Scope Without Braces
**ZAKAZANE** - Wszystkie control statements **MUSZĄ** mieć `{}`.

```csharp
// ❌ NIGDY NIE UŻYWAJ
if (condition) return value;
if (data == null) DoSomething();
for (int i = 0; i < 10; i++) Process(i);

// ✅ ZAWSZE UŻYWAJ
if (condition)
{
    return value;
}

if (data == null)
{
    DoSomething();
}

for (int i = 0; i < 10; i++)
{
    Process(i);
}
```

**Powody:**
- Zapobiega błędom przy dodawaniu kodu
- Spójność i czytelność
- Zgodność z enterprise standards

---

### 3. Inline Object Passing
**ZAKAZANE** - Zawsze twórz zmienną lokalną przed przekazaniem do metody.

```csharp
// ❌ NIGDY NIE UŻYWAJ
return Ok(products.Select(p => (ProductResponse)p));
return NotFound(new { message = "Error" });
return CreatedAtAction(nameof(GetById), new { id = created.Id }, (ProductResponse)created);

// ✅ ZAWSZE UŻYWAJ
var responses = products.Select(p => (ProductResponse)p);
return Ok(responses);

var errorResponse = new { message = "Error", };
return NotFound(errorResponse);

var routeValues = new { id = created.Id, };
var response = (ProductResponse)created;
return CreatedAtAction(nameof(GetById), routeValues, response);
```

**Powody:**
- Łatwiejsze debugowanie (breakpoints, inspect)
- Możliwość sprawdzenia wartości przed przekazaniem
- Lepsza czytelność

---

### 4. Missing Trailing Commas
**ZAKAZANE** - Brak trailing comma w ostatnim elemencie.

```csharp
// ❌ NIGDY NIE UŻYWAJ
var product = new Product
{
    Name = "Laptop",
    Price = 1999.99m
};

// ✅ ZAWSZE UŻYWAJ
var product = new Product
{
    Name = "Laptop",
    Price = 1999.99m,  // ← trailing comma
};
```

**Powody:**
- Czysty git diff przy dodawaniu nowych properties
- Mniej merge conflicts
- Zgodne z TypeScript/JSON best practices

---

### 5. Missing Blank Lines
**ZAKAZANE** - Brak pustych linii przed control statements i return.

```csharp
// ❌ NIGDY NIE UŻYWAJ
var data = await GetDataAsync();
if (data == null)
{
    return NotFound();
}
_logger.LogInformation("Processing");
return Ok(result);

// ✅ ZAWSZE UŻYWAJ
var data = await GetDataAsync();

if (data == null)
{
    return NotFound();
}

_logger.LogInformation("Processing");

return Ok(result);
```

**Powody:**
- Czytelność - wizualne oddzielenie logiki
- Łatwiejsze skanowanie kodu
- Spójność z enterprise standards

### 6. Namespace Position
**ZAKAZANE** - `using` dyrektywy przed `namespace`.

```csharp
// ❌ NIGDY NIE UŻYWAJ
using System;
using Microsoft.EntityFrameworkCore;

namespace DAL.Data;

public class AppDbContext : DbContext { }

// ✅ ZAWSZE UŻYWAJ
namespace DAL.Data;

using System;
using Microsoft.EntityFrameworkCore;

// ✅ WYJĄTEK: nullability (nullable enable/disable)
```

**Powody:**
- Namespace identyfikuje plik od razu (pierwsza linia)
- File-scoped namespace style (modern C#)
- Spójność w całym solution
- Wyjątek tylko dla nullability directives

### 3. Validation - Configurable Business Rules
**DataAnnotations są ZABRONIONE** - walidacja przez **FluentValidation** z konfiguracją.

```csharp
// ❌ NIGDY - DataAnnotations
public sealed record CreateProductRequest
{
    [Required(ErrorMessage = "Nazwa produktu jest wymagana")]
    [StringLength(200, MinimumLength = 2)]
    public required string Name { get; init; }
}

// ✅ ZAWSZE - Czysty model
public sealed record CreateProduct
{
    public required string Name { get; init; }
    public required decimal Price { get; init; }
}

// ✅ Konfiguracja w appsettings.json
{
  "Validation": {
    "Product": {
      "NameMinLength": 2,
      "NameMaxLength": 200,
      "PriceMin": 0.01,
      "PriceMax": 999999.99
    }
  }
}

// ✅ FluentValidation - deklaratywny walidator
internal sealed class CreateProductValidator : AbstractValidator<CreateProduct>
{
    public CreateProductValidator(ValidationConfig config)
    {
        var rules = config.Product;

        RuleFor(x => x.Name)
            .NotEmpty()
            .WithMessage("Nazwa produktu jest wymagana")
            .Length(rules.NameMinLength, rules.NameMaxLength)
            .WithMessage($"Nazwa musi mieć od {rules.NameMinLength} do {rules.NameMaxLength} znaków");

        RuleFor(x => x.Price)
            .InclusiveBetween(rules.PriceMin, rules.PriceMax)
            .WithMessage($"Cena musi być między {rules.PriceMin} a {rules.PriceMax}");

        RuleFor(x => x.StockQuantity)
            .GreaterThanOrEqualTo(0)
            .WithMessage("Ilość nie może być ujemna");
    }
}

// ✅ Service zwraca validation result
public async Task<(ValidationResult Validation, Product? Product)> CreateProductAsync(CreateProduct model)
{
    var fluentResult = await _createValidator.ValidateAsync(model);
    var validation = ValidationResult.FromFluentValidation(fluentResult);

    if (!validation.IsValid)
    {
        return (validation, null);
    }

    var entity = (ProductEntity)model;
    var created = await _repository.AddAsync(entity);
    var result = (Product)created;

    return (validation, result);
}

// ✅ Controller obsługuje validation
var (validation, product) = await _productService.CreateProductAsync(model);

if (!validation.IsValid)
{
    var errors = validation.Errors.ToDictionary(e => e.Field, e => e.Message);

    return BadRequest(new
    {
        message = "Walidacja nie powiodła się",
        errors = errors,
    });
}
```

**Powody:**
- **FluentValidation** - czytelne, deklaratywne reguły
- Reguły biznesowe w konfiguracji (JSON), nie w kodzie
- Zmiana walidacji bez rekompiacji
- Testowalna logika walidacji
- Separacja techniczna/biznesowa
- Brak magicznych atrybutów
- Kontrola nad komunikatami błędów
- `async` validation support

**Struktura:**
```
BLL/
  Validation/
    ValidationError.cs          ← sealed record
    ValidationResult.cs         ← sealed record + FromFluentValidation()
    ValidationConfig.cs         ← config class z ProductValidationRules
    CreateProductValidator.cs   ← AbstractValidator<CreateProduct>
    UpdateProductValidator.cs   ← AbstractValidator<UpdateProduct>
```

**DI Registration:**
```csharp
services.AddScoped<IValidator<CreateProduct>>(sp =>
{
    var config = sp.GetRequiredService<ValidationConfig>();
    return new CreateProductValidator(config);
});
```

### 6. Repository Pattern

```csharp
// Baza abstrakcyjna dla generic repository
internal abstract class Repository<T> : IRepository<T> where T : class
{
    protected readonly AppDbContext _context;
    protected readonly DbSet<T> _dbSet;

    protected Repository(AppDbContext context)
    {
        _context = context;
        _dbSet = context.Set<T>();
    }

    public async Task<T?> GetByIdAsync(int id)
    {
        return await _dbSet.FindAsync(id);
    }

    public async Task<T> AddAsync(T entity)
    {
        var entry = await _dbSet.AddAsync(entity);

        await _context.SaveChangesAsync();

        return entry.Entity;
    }

    public async Task DeleteAsync(int id)
    {
        var entity = await _dbSet.FindAsync(id);

        if (entity != null)
        {
            _dbSet.Remove(entity);
            await _context.SaveChangesAsync();
        }
    }
}

// Konkretna implementacja
internal sealed class ProductRepository : Repository<Product>, IProductRepository
{
    public ProductRepository(AppDbContext context) : base(context) { }

    public async Task<IEnumerable<Product>> GetByCategoryAsync(string category)
    {
        return await _dbSet
            .Where(p => p.Category.ToLower() == category.ToLower())
            .OrderBy(p => p.Name)
            .ToListAsync();
    }
}
```

### 7. Visibility & Access Control

**ZASADY KONTROLI DOSTĘPNOŚCI:**

#### `internal` - Implementation Details
Wszystkie **implementacje** services i repositories są `internal`:

```csharp
// ✅ BLL/Services/ProductService.cs
internal sealed class ProductService : IProductService { }

// ✅ DAL/Repositories/ProductRepository.cs
internal sealed class ProductRepository : Repository<Product>, IProductRepository { }

// ✅ DAL/Repositories/Repository.cs (base class)
internal abstract class Repository<T> : IRepository<T> where T : class { }
```

**Powody:**
- Ukrycie implementacji przed zewnętrznymi projektami
- Dostęp tylko przez interfejsy
- Kontrola API surface

#### `public` - Public Contracts
Tylko **interfejsy** i **DTOs/Models** są `public`:

```csharp
// ✅ BLL/Services/IProductService.cs
public interface IProductService { }

// ✅ BLL/DTOs/ProductDto.cs
public class ProductDto { }

// ✅ WebAPI/Models/ProductResponse.cs
public sealed record ProductResponse { }

// ✅ DAL/Repositories/IProductRepository.cs
public interface IProductRepository : IRepository<Product> { }

// ✅ DAL/Entities/Product.cs
public class Product { }
```

**Powody:**
- Jasny public API
- Dependency Injection działa przez interfejsy
- DTOs/Models są kontraktami warstwy

#### `sealed` - No Inheritance
Wszystkie **concrete classes** (nie bazy) są `sealed`:

```csharp
// ✅ Services
internal sealed class ProductService : IProductService { }

// ✅ Repositories
internal sealed class ProductRepository : Repository<Product>, IProductRepository { }

// ✅ DbContext
public sealed class AppDbContext : DbContext { }

// ✅ Middleware
public sealed class ExceptionHandlingMiddleware { }

// ✅ Program
public sealed class Program
{
    private Program() { }
}

// ✅ Records/Models
public sealed record ProductResponse { }
public sealed record CreateProductRequest { }
```

**Powody:**
- Performance (compiler optimization)
- Intencja - klasa nie jest bazą
- Zapobiega niezamierzonemu dziedziczeniu
- Ułatwia refactoring

#### `abstract` - Base Classes Only
Tylko **bazy generyczne** są `abstract`:

```csharp
// ✅ Generic repository base
internal abstract class Repository<T> : IRepository<T> where T : class
{
    protected readonly AppDbContext _context;
    protected readonly DbSet<T> _dbSet;

    protected Repository(AppDbContext context)
    {
        _context = context;
        _dbSet = context.Set<T>();
    }
}
```

**Powody:**
- Wspólna funkcjonalność dla różnych typów
- Nie może być instancjonowana bezpośrednio
- Protected constructor dla klas pochodnych

#### Podsumowanie Reguł

| Typ | Visibility | sealed/abstract | Przykład |
|-----|-----------|-----------------|----------|
| **Interface** | `public` | N/A | `IProductService`, `IRepository<T>` |
| **Model (BLL)** | `public` | `sealed` | `Product`, `CreateProduct` |
| **Model (WebAPI)** | `public` | `sealed` | `ProductResponse`, `CreateProductRequest` |
| **Entity** | `public` | - | `Product` (DAL.Entities) |
| **Service Implementation** | `internal` | `sealed` | `ProductService` |
| **Repository Implementation** | `internal` | `sealed` | `ProductRepository` |
| **Generic Base** | `internal` | `abstract` | `Repository<T>` |
| **DbContext** | `public` | `sealed` | `AppDbContext` |
| **Middleware** | `public` | `sealed` | `ExceptionHandlingMiddleware` |
| **Program** | `public` | `sealed` | `Program` (+ private ctor) |

### 10. Program.cs Pattern

```csharp
public sealed class Program

    }

        await app.RunAsync();
    }
```

## 📝 Naming Conventions

### Variables
- `model` - dla BLL models (zamiast `dto`)
- `result` - dla wartości zwracanych (output)
- `entity` - dla DAL entities
- `response` - dla API responses
- `request` - dla API requests
- `models` / `responses` - dla kolekcji

### Methods
- `async` suffix dla asynchronicznych metod
- `Get*Async` - pobieranie danych
- `Create*Async` - tworzenie
- `Update*Async` - aktualizacja
- `Delete*Async` - usuwanie
- `*Exists` - sprawdzanie istnienia

### Classes
- `*Controller` - kontrolery
- `*Service` - serwisy
- `*Repository` - repozytoria
- `*Request` - API request models (WebAPI.Models)
- `*Response` - API response models (WebAPI.Models)
- `Create*` / `Update*` - command models (BLL.Models)
- `*` - query/read models (BLL.Models, np. `Product`)
- `*Entity` / `*` - database entities (DAL.Entities)
